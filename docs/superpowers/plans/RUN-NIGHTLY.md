# Что делать

1. Прочитай docs/superpowers/plans/MANIFEST.md из корня репозитория.

2. Найди ПЕРВЫЙ стейдж со статусом `- [ ]` (todo) по порядку.

3. Проверь, что его имени соответствующего открытого PR НЕТ:
      gh pr list --state open --search "[Stage N]"
   (где N — номер стейджа, подставляй реальный)
   Если PR уже открыт — СКИПАЙ этот стейдж, пиши отчёт
   "Stage N has an open PR awaiting merge" и выходи. НЕ трогай его.

4. Если нет открытого PR — проверь, не смержена ли его ветка:
      gh pr list --state merged --search "[Stage N]"
   Если смержена но MANIFEST всё ещё `- [ ]` — это рассинхронизация,
   STOP с отчётом (я разберусь утром).

5. Если всё чисто — приступай к исполнению:

   a. Прочитай план этого стейджа целиком. Файл указан в MANIFEST.
   b. Создай рабочую ветку с префиксом `claude/`:
         git checkout -b claude/stage-N-<short-name>
      (короткое имя возьми из названия плана)
   c. Обнови MANIFEST.md: статус `- [ ]` → `- [~]` (первым коммитом
      на этой ветке):
         git commit -m "chore(manifest): mark stage-N in progress"
   d. Выполни план через skill `superpowers:subagent-driven-development`:
      fresh subagent на Task, two-stage review, коммиты по шагам как
      в плане. Сообщения коммитов — из плана.
   e. После всех Tasks прогоняй acceptance:
         go build ./... && go vet ./... && go test ./...
      Все exit 0. Иначе STOP.
   f. Сверь с "Definition of Done" плана. Каждый пункт — check.
   g. Обнови MANIFEST.md: статус `- [~]` → `- [x]` (последним коммитом):
         git commit -m "chore(manifest): mark stage-N done"
   h. Запушь ветку:
         git push -u origin claude/stage-N-<short-name>
   i. Открой PR:
         gh pr create --base main --head claude/stage-N-<short-name> \
           --title "[Stage N] <название из плана>" \
           --body-file <(cat <<'EOF'
         ## Stage N — <название>

         План: docs/superpowers/plans/<plan-file>.md

         ### Acceptance evidence
         - build: ✅
         - vet: ✅
         - test: ✅
         - Definition of Done checked: ✅

         ### Tasks completed
         <список Tasks с коротким описанием что сделано>

         ### Что было необычного
         <1-5 строк. Если всё гладко — "ничего, прошло по плану">

         🤖 Generated by scheduled agent
         EOF
         )
   j. Напиши docs/superpowers/plans/NIGHT-RUN-REPORT.md на этой же
      ветке (НЕ на main) — см. "Формат успешного отчёта".
   k. Последний коммит и push этого отчёта:
         git add docs/superpowers/plans/NIGHT-RUN-REPORT.md
         git commit -m "docs: night run report for stage-N"
         git push
   l. ВЫХОДИ. НЕ переходи к следующему стейджу.

# STOP-условия

Немедленно останови работу если:
- build / vet / test красные и не чинятся за 2 попытки
- План расходится с реальностью, нужно править план
- Docker недоступен (проверь: `docker info` должен работать)
- Network-policy блокирует образ MariaDB — тесты не запустятся
- Публичное v1-поведение сломано неожиданно
- gh / git отказывается работать (проблема с credentials)
- Stage 4: policy assumptions конфликтуют с кодом

При STOP:
  1. НЕ пытайся доделать через shortcut (игнор теста, --no-verify и т.п.)
  2. Закоммить то что работает, запушь ветку
  3. Открой PR с меткой "[Stage N][STOPPED]" в заголовке
  4. В теле PR опиши где застрял по шаблону ниже
  5. Напиши NIGHT-RUN-REPORT.md на ветке по шаблону STOPPED
  6. Запушь отчёт, выходи

# Жёсткие правила

- НИКОГДА --no-verify, --no-gpg-sign
- НИКОГДА git push --force
- НИКОГДА не пуши в main напрямую — только через PR
- НИКОГДА не мержи PR сам (это делает человек утром)
- НИКОГДА не амендь коммиты — только новые
- НИКОГДА не меняй версии зависимостей кроме указанных `go get` из плана
- НИКОГДА не удаляй тесты чтобы сделать зелёным
- НИКОГДА не скипай шаги плана
- НИКОГДА не исполняй больше одного стейджа за запуск
- НЕ трогай файлы других стейджей — только твоего
- Branch ALWAYS starts with `claude/` — иначе push будет отвергнут
  remote'ом по политике routine

# Проверка окружения (в самом начале)

Прогони до того как брать стейдж:
  git branch --show-current           # должен быть main
  git status --porcelain               # должен быть пустой
  docker info 2>&1 | head -1           # Docker работает?
  gh auth status                       # gh аутентифицирован?
  go version                           # Go 1.24+?
Если что-то из этого не работает — STOP, НЕ пытайся чинить окружение.

# Формат успешного отчёта

docs/superpowers/plans/NIGHT-RUN-REPORT.md (на твоей ветке):

  # Night Run Report — YYYY-MM-DD

  ## Результат: SUCCESS

  ## Стейдж: stage-N — <название>

  ## Ссылки
  - PR: <url из `gh pr create` output>
  - Branch: claude/stage-N-<short-name>
  - Session transcript: https://claude.ai/code/routines (смотреть текущий run)

  ## Сделано
  - коммитов на ветке: N
  - acceptance: build ✅ vet ✅ test ✅
  - DoD: все пункты плана отмечены

  ## Что было необычного
  <1-5 строк>

  ## Что делать человеку утром
  1. Прочитать PR описание
  2. Прогнать глазами diff
  3. Смержить PR (merge commit, не squash — чтобы сохранить историю Tasks)
  4. После merge: MANIFEST в main автоматически обновится (он в PR)
  5. Следующая ночь подхватит stage-(N+1)

# Если следующий стейдж помечен `- [~]` в main

Это бага модели — такого быть не должно. Мы обновляем `- [~]` только
на ветке, в main идёт сразу `- [x]` после merge. Если видишь `- [~]`
в main:
  Результат: SKIPPED
  Причина: MANIFEST в main has stage-N as `- [~]` — rogue state.
  Needs manual cleanup.

## После каждой ночи (твой утренний ритуал)

1. Проверь [claude.ai/code/routines](https://claude.ai/code/routines) — читай transcript последнего run'а
2. Проверь PRs на GitHub: должен быть один новый `[Stage N] ...`
3. Прочитай тело PR — агент приложил acceptance evidence
4. Сверь diff (особенно новые файлы, сигнатуры сервисов, удалённые тесты если есть)
5. Если всё ок — **merge commit** (не squash, не rebase). После merge:
   - MANIFEST в main обновится (помечен `- [x]`)
   - Следующая ночь подхватит следующий стейдж
6. Если нужны правки — запроси у агента changes через commit в PR, либо правь сам, потом merge
7. Если STOPPED PR — разбирайся с причиной:
   - Поправь план или код
   - Либо закрой PR и сделай стейдж вручную
   - Либо исправь что мешало и перезапусти routine (агент увидит что PR закрыт, попробует снова)
